# xiaok Slash 菜单稳定性修复设计

**日期：** 2026-03-30
**状态：** 已确认，待实现

---

## 概述

本设计解决 `xiaok` 交互模式下 slash 菜单的两个核心问题：

1. 输入 `/` 后菜单虽然被逻辑正确触发，但终端上显示位置不稳定。
2. 菜单打开后光标可能漂移，导致输入区、菜单区和后续输出相互干扰。

本次改动的目标不是复制 Claude 的整套终端视觉，而是优先保证功能正确：slash 菜单必须稳定显示、键盘操作必须可靠、后续视觉优化必须建立在可验证的终端行为之上。

---

## 问题分析

### 当前现象

- 输入 `/` 时，`input.ts` 中的过滤与菜单渲染逻辑已经执行。
- 终端中仍会出现菜单显示异常、光标跑偏、菜单区域与输入区互相覆盖的问题。

### 根因

根因不是 slash 菜单过滤逻辑，而是终端布局控制分散在多个模块：

- [`src/commands/chat.ts`](/Users/song/projects/xiaok-cli/src/commands/chat.ts) 在交互循环里使用绝对定位和底部输入区假设。
- [`src/ui/input.ts`](/Users/song/projects/xiaok-cli/src/ui/input.ts) 以“当前输入行下面还能继续打印菜单”为前提进行渲染和清理。
- [`src/ui/statusbar.ts`](/Users/song/projects/xiaok-cli/src/ui/statusbar.ts) 已经改成普通文本输出，但 `chat.ts` 仍保留底部输入区相关 ANSI 控制。

当 `chat.ts` 把输入区固定在终端底部附近时，`input.ts` 再向下打印多行菜单，就会和底部空间、滚动区、光标恢复行为发生冲突。

---

## 目标

- 输入 `/` 后，菜单稳定显示在当前输入行下方。
- 菜单打开、更新、关闭时，光标位置始终与输入内容一致。
- 上下键在菜单打开时只切换选项，不触发历史输入导航。
- Enter 和 Tab 只对当前选中项做补全/确认，不破坏输入行。
- 终端 resize、`/clear`、普通消息输出不会重新引入光标错位。

## 非目标

- 本次不实现完全仿 Claude 的底部固定输入框。
- 本次不重做 welcome screen 的视觉风格。
- 本次不做复杂的弹层、浮层或“向上展开”菜单。

---

## 方案比较

### 方案 A：回到正常文档流输入区

做法：

- 移除交互循环中对输入区的底部绝对定位和滚动区域控制。
- 每轮输入前直接按文档流打印分隔线、提示符和输入框。
- slash 菜单继续由 `input.ts` 在输入行下方渲染。

优点：

- 与当前 `input.ts` 的菜单实现天然兼容。
- ANSI 冲突最少，终端兼容性最好。
- 修复面最小，最容易用测试固定行为。

缺点：

- 输入区不会固定在屏幕底部。
- 视觉上和 Claude 截图仍有差距。

### 方案 B：保留现有输入区视觉，但取消底部固定定位

做法：

- 仍保留分隔线、提示符等视觉结构。
- 输入区和菜单作为连续内容块输出。
- `chat.ts` 不再抢占终端底部，只负责顺序打印。

优点：

- 能保留部分现有观感。
- 仍然避免了底部定位冲突。

缺点：

- 需要更仔细地梳理 `chat.ts` 和 `input.ts` 的职责边界。
- 本质上仍接近方案 A，只是文案和渲染层次稍复杂。

### 方案 C：保留底部固定输入区，重写菜单定位

做法：

- 保留底部固定输入区。
- 将 slash 菜单改成向上展开或独立管理绝对定位区域。

优点：

- 最接近 Claude 的交互形态。

缺点：

- ANSI 控制最复杂，最容易继续出现光标错位和清理残影。
- resize、clear、长输出、多轮对话都需要额外边界处理。
- 不适合作为第一轮稳定性修复。

### 推荐方案

采用 **方案 A** 作为第一阶段实现，必要时吸收方案 B 的部分视觉结构，但明确放弃底部固定输入区。

理由：

- 用户已经明确要求“先保证功能的正确，不要再出问题”。
- 当前 bug 的根源就是多个模块同时控制终端定位；继续保留底部固定输入区会扩大不确定性。
- 只有先把布局控制权收敛到单一路径，后续才有资格做视觉优化。

## 设计

### 1. 交互布局原则

交互模式回到正常文档流：

- welcome screen 正常打印并随滚动自然上移。
- 每轮输入前由 `chat.ts` 顺序打印分隔线和提示符。
- `input.ts` 负责输入过程中的单行重绘和菜单渲染。
- 状态栏继续作为普通文本行输出，不做固定底部渲染。

### 2. 模块职责

#### `chat.ts`

- 负责交互循环和消息顺序输出。
- 不再写入底部滚动区域 ANSI。
- 不再在每轮输入前把光标强行移动到倒数第 2 或倒数第 3 行。
- `resize` 处理不再重绘底部状态栏，而是降级为无操作或仅保留安全行为。

#### `input.ts`

- 继续维护输入内容、光标位置、菜单项和菜单索引。
- 菜单只相对当前输入行渲染和清理。
- 菜单打开时，上下键优先服务菜单；菜单关闭时才恢复历史输入导航。
- 菜单清理后，输入行必须恢复到正确光标位置。

#### `statusbar.ts`

- 保持当前“普通文本输出”的策略。
- 不重新引入绝对定位、保存/恢复光标、滚动区域。

## 具体改动

### `src/commands/chat.ts`

- 删除交互循环中的底部定位输出：
  - 倒数第 3 行分隔线定位
  - 倒数第 2 行输入提示定位
  - 用户提交后跳回滚动区域最后一行的定位
- 删除或弱化 `handleResize()` 中依赖底部状态栏的逻辑。
- `/clear` 后重新打印欢迎界面和后续输入提示，但不再重建底部固定布局。

### `src/ui/input.ts`

- 保留现有 slash 菜单的核心过滤逻辑。
- 校正菜单渲染、清理和输入行恢复顺序，确保其建立在普通文档流之上。
- 需要时补一个内部辅助函数，让“菜单清理后重画输入行”行为更明确，减少状态分散。

### 测试

新增或增强以下测试：

- `tests/ui/input.test.ts`
  - 输入 `/` 时能触发菜单输出。
  - 菜单清理后不会破坏输入行内容。
  - 菜单打开时上下键切换选项，关闭时才进入历史导航。
- `tests/commands/chat.test.ts` 或同类测试文件
  - 交互模式不再输出底部定位 ANSI。
  - `/clear` 后仍能重新进入可用输入状态。

如果当前缺少 `chat.ts` 的交互测试文件，本次允许补一个最小回归测试文件，只覆盖这次去除底部定位的行为。

## 错误处理与边界

- 当菜单项为空时，不渲染菜单，不保留菜单状态。
- 当输入从 `/` 变为普通文本时，立即关闭菜单并恢复单行输入状态。
- 当用户按 `Esc` 关闭菜单时，只关闭菜单，不清空输入内容。
- 当终端宽度变化时，不尝试做复杂的动态重定位；允许下一次重绘自然适配新宽度。

---

## 验证策略

实现必须按以下顺序验证：

1. 先写失败测试，证明当前交互模式仍依赖底部定位或菜单恢复行为不稳定。
2. 实现最小修复，只收敛终端定位控制，不顺手做视觉重构。
3. 跑相关 UI 测试，确认菜单打开、移动、确认、关闭都稳定。
4. 手动验证：
   - 输入 `/`
   - 输入 `/br`
   - 上下键切换
   - Tab 补全
   - Enter 选中
   - Esc 关闭
   - `/clear`
   - 普通提问后再次输入 `/`
