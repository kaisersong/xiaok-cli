🟠 High 级（独立发现）

::: callout warning

属性值
发现者claude
位置src/ai/tools/bash-safety.ts
维度安全性

描述：classifyBashCommand() 使用纯正则匹配判定命令风险，存在 4 种绕过向量：

  1. Base64 编码绕过：echo "cm0gLXJmIC8=" | base64 -d | sh
  2. Unicode/混淆绕过：全角字符、零宽空格、${IFS} 变量展开
  3. 间接执行绕过：python3 -c "import os; os.system('rm -rf /')"
  4. 变量拼接绕过：X=su; Y=do; $X$Y rm -rf /

修复指引：

  • 增加 base64/hex 编码检测
  • 增加解释器间接执行检测（python/ruby/perl）
  • 考虑引入 AST 级命令解析替代纯正则

::: callout warning

属性值
发现者claude
位置src/ai/permissions/workspace.ts
维度安全性

描述：assertWorkspacePath() 使用字符串前缀匹配验证路径，未调用 realpathSync。攻击者可在工作区内创建符号链接指向外部敏感文件（如 /etc/passwd），通过 write/edit 工具操作符号链接即可读写工作区外任意文件。

修复指引：

  • 在路径校验前调用 realpathSync 解析符号链接
  • 对符号链接本身是否指向工作区外做额外检查

包含模型适配、权限管理、skill 编排、UI 渲染、会话管理等十余种职责。仅 4 个 export、14 个 function，大量逻辑以闭包形式存在。建议按职责拆分为 4-5 个模块。

发现者qoder位置layered-store.ts（13 处）、compaction.ts（4 处）

.all() 返回 unknown 类型，统一使用 as any[] 绕过类型检查。修改数据库 schema 时编译器无法检测下游影响。建议为每个查询结果定义显式类型接口。

::: callout info

发现者claude位置bash.ts, write.ts, read.ts, edit.ts

所有工具仅依赖 TypeScript 类型断言，无运行时 JSON Schema 验证。AI 模型可能产生类型错误的参数，MCP 外部工具可能传入意外类型。建议引入 zod/ajv 运行时校验层。

::: callout info

发现者qoder位置transcript.ts:79, client.ts:36

来自文件系统和外部进程的 JSON 直接 parse，格式错误时会导致未捕获异常。建议对所有 JSON.parse 调用添加 try-catch，特别是输入来自网络或文件系统的场景。

::: callout info

发现者qoder位置src/platform/mcp/transport.ts:162-203

连接断开后不会自动重连，无 keep-alive 机制，连接超时无保护。建议添加心跳 ping-pong、指数退避重连和连接超时。

::: callout info

发现者qoder位置package.json

@anthropic-ai/sdk 从 0.39.0 到最新 0.102.0（63 个 minor 版本差距）、openai 从 4.77.0 到 6.42.0（2 个大版本）、commander 从 12.0.0 到 15.0.0。建议制定依赖升级计划，按风险分批进行。

::: callout info

发现者claude位置src/runtime/daemon/launcher.ts

spawnXiaokDaemonDetached() 启动后立即 unref()，不监控进程状态。daemon 崩溃后定时任务停止执行而无通知。建议注册 exit 监听器、实现 watchdog 模式。

::: callout info

发现者xiaok位置desktop/electron/preload.cjs

通过 contextBridge 暴露约 120+ 方法到渲染进程，包括模型管理、任务控制、文件系统操作、KSwarm 控制等。建议审计并移除渲染进程不需要的 API，添加操作权限校验。

::: callout info

发现者xiaok位置package.json vs desktop/package.json

CLI 和 Desktop 使用不同版本号，用户难以追踪版本对应关系。建议使用单一版本源或在 Desktop 版本号中加入 CLI 版本信息。

::: callout info

发现者xiaok位置desktop/renderer/src/components/ChatShell.tsx

无骨架屏、进度指示器或超时处理，长时间运行操作（可达 30 分钟）不展示中间进度。建议添加骨架屏、流式响应动画指示器、超时提示和重试按钮。

::: callout info

发现者xiaok位置desktop/electron/desktop-services.ts:86-189

降级到 JSON 文件存储只打了一条 console.warn，用户无感知。建议通过 IPC 通知渲染进程存储引擎状态，添加原生模块健康检查。

所有服务通过闭包共享状态，单元测试极难编写。建议为每个服务定义 TypeScript 接口，使用工厂函数接受接口。

质量原则

从本次交叉评审中归纳的 5 条可复用质量原则：

原则 1: 纵深防御 — 安全不可依赖单一机制

来源：QF-CL02/X03（sandbox）、QF-X01（CSP）、QF-CL01（bash regex）、QF-CL08（symlink）

即使 contextIsolation: true + nodeIntegration: false 已配置，仍需要 sandbox、CSP、realpath 校验等多层防护。任何单一安全机制的假设都可能被绕过。bash 安全分类不能仅依赖正则——需要 AST 解析或沙箱执行作为第二层。

原则 2: 模块边界 — 单体文件是技术债务的温床

来源：QF-CL04/X02（desktop-services 5400+ 行）、QF-Q01（chat.ts 3511 行）

超过 1000 行的单体文件应触发架构审视。职责混杂导致测试困难、修改风险高、新人上手慢。在模块达到 500 行时即应考虑职责边界和拆分策略。

原则 3: 错误透明 — 绝不静默吞没异常

来源：QF-Q05（30 处空 catch）、QF-X07（sqlite 静默降级）、QF-Q06（JSON.parse 无保护）

空 catch 块和静默降级在生产环境中是调试黑洞。所有异常至少应记录结构化日志；关键路径降级（如存储引擎）应通知用户并提供恢复指引。ESLint no-empty 规则应禁止空 catch 块。

原则 4: 依赖卫生 — 定期审计是安全基线

来源：QF-Q03（vitest critical CVE）、QF-Q04（ws 版本不匹配）、QF-Q11（多依赖落后）

依赖管理不是一次性工作。应建立：① CI 中的 npm audit 阻断 critical/high；② lockfile 版本一致性校验；③ 季度依赖升级计划。vitest 3.2.4 → 3.2.6 仅差 2 个 patch 版本却存在关键漏洞，说明 patch 版本更新不可忽视。

原则 5: 防御性接口 — 信任但要验证

来源：QF-CL03（工具输入无运行时验证）、QF-X04（preload API 表面过大）、QF-CL08（workspace 路径校验）

外部输入（AI 模型、MCP 外部工具、用户文件路径）不能假设其合法性。工具输入应进行 JSON Schema 运行时验证；preload 暴露的 API 应做权限校验和频率限制；文件路径操作前应解析符号链接。TypeScript 类型系统是编译时保障，不是运行时安全。
